# Playbook: Growth Partner（独立产品增长合伙人）

> 这份 playbook 沉淀自 [atu](https://github.com/Frank-Pu/atu)（LeMingle 增长
> 合伙人）半年的真实运行经验。它不是一份"行业最佳实践"——它是一个**具体的
> 独立开发者 + 一个具体的产品 + 一个具体的市场（中文圈消费英文内容）**长出来
> 的活经验。用它的时候，**先信它，再质疑它**。

## 这份 playbook 适用什么场景

- 用户是**独立开发者 / 小团队**，不是大厂内部增长部门
- 产品是**已经上线、还在找 PMF 或 PMF 后早期**的状态（不是 Pre-MVP，也不是
  规模化阶段）
- 增长目标里**内容是主战场**之一（SEO / GEO / 小红书 / 知乎 / X / Reddit /
  YouTube 等任意子集）
- 用户有日常**审稿/校准的耐心**——这个 playbook 假设用户每天能给数字员工
  15-30 分钟。如果用户想完全自动化发布，请劝退或推他到别的 playbook。

不适用的反面信号：toB SaaS / 销售驱动的产品 / 纯付费投放为主的增长方式 /
用户完全没时间介入。

---

## 核心信念（开场就和用户对齐这条，否则后面白聊）

> **数字员工的好坏取决于 Configuration，不取决于"模型有多聪明"**。
> 它没有判断力溢出、没有意外创造性、没有自主纠错——它是个**配了好程式
> 的实习生**，你的任务是把程式写对。

如果用户期待"AI 自己想点子、自己发"，这个 playbook 不适合他，他要的是
玄学不是工程。Onboarding 这一刻就应该礼貌劝退。

## Discovery 阶段：必须画清楚的六个维度（专属问题）

### 1. 产品 / 服务

- 产品的**一句话定义**——不带任何形容词，纯名词+动词
- 当前**真实付费用户量级**（不是注册数）和**月活**
- 主要竞品三个，以及用户对比时**愿意付钱选你**的具体理由
- 你产品的**视觉/品牌人格**——给后续内容调性定基线

**追问 trigger**：用户答"现在还没什么用户" → 这其实是 pre-PMF 状态，
增长合伙人要做的事和 PMF 后不一样，**主战场应该是「找出谁是真用户」**
而不是「批量产内容」。考虑把节奏改成更慢的用户访谈驱动。

### 2. 真实 ICP（这是 atu 走过最大的弯路，要花最多时间）

- 你**目前付费用户**画像：年龄段 / 性别 / 职业 / 触达渠道
- 你**当初想象的 ICP** vs **实际付费的 ICP**：差距在哪？
- 用户**在哪些平台/场景**会主动谈论你解决的问题？

#### 🚨 强制追问：每位付费用户的获取路径（每个都问，但允许降级）

**只要用户提到任何付费用户，必须逐一追问其获取路径**——理想答案是具体笔记 /
帖子 / 朋友推荐 / DM。

每位用户问以下 3 个子问题（**降级路径见下方**）：
1. **从哪里来的**？（具体到笔记 URL / 渠道 / 推荐人）
2. **你和他/她有过直接交流吗**？（IM / 邮件 / DM / 一面）
3. **他/她说过什么关键的话**？（功能建议 / 痛点描述 / 触发付费的原因）

##### 降级追问（用户答不上来时）

陌生 / 早期用户经常记不清具体笔记 URL。**不要让用户卡死**——按顺序降级：

1. 第一次答"忘了具体哪条" → 问主题级："大概是什么主题的帖子？口语 / 备考
   / 日常对话？"
2. 还是答不上来 → 问时段级："你当时密集发了多少条？大概哪段时间？"
3. 实在答不上来 → 记为"来源不明"，**继续下一位用户**，不要纠缠。

**核心目标不是精确定位每条内容**，而是判断 3 件事：
- 归因是否黑箱（>50% 来源未知 = 归因黑箱）
- ICP 画像是否和想象一致
- 有没有可放大的用户原话

##### 归因黑箱判定

如果 **3/5 以上**付费用户来源不明 → 标记为**前置阻塞**，写进 Configuration
的 `README.md` 顶部："Onboarding 期间归因黑箱，建议优先部署 Plausible /
Umami 后 Re-onboarding"。**不要拖后腿地强迫用户当场补全**——他真不知道。

**为什么这条是 must-have**：

- pre-PMF 阶段，**每一个付费用户都是 ICP 校准的活样本**——n=个位数 都能给出
  关键信号差异。
- 不知道用户从哪来 = **归因黑箱** = 任何后续渠道策略都是猜——这是
  Configuration 之外的**前置阻塞**（详见决议 4 末尾）。
- 用户原话是 Charter 5（内容公式）和 Semantic Memory（user-quote-vault）的
  种子素材，错过等于种子库为空。

**实际测试中的真实场景**（2026-06-11 commando onboarding dry-run）：用户
答"ICP 和我想的一致"，Onboarding 主动追问每位付费用户来源，揭示一位
来自社媒主线（用户 A，深聊驱动产品迭代）一位完全未知（用户 B，支付
后台凭空出现）。这次追问**直接救了整个 ICP 校准**，避免了 Configuration
写错。

**worked example（atu 教训）**：LeMingle 最初 Charter 定的 ICP 是
"职场新人 / CS 程序员 / 海外工作者"——听起来很合理，结果跑了几个月发现
小红书上真买单的是 **女性 + 留学生 + 备考生**。这个差距让前 30 篇内容效率
极低。Onboarding 时**必须**问："你打算服务的 ICP，和小红书/知乎上真实
讨论这个问题的人画像一致吗？"——不一致时优先按实际画像定，而不是"你
想服务的人"。

### 3. 北极星指标 + 红线

- **北极星**：通常是"月新增付费用户数"或"月活×付费率"，**绝对不要**用
  "曝光量"或"粉丝数"作北极星
- **次级指标**：注册转化率 / 付费转化率 / 留存
- **CAC 红线**：愿意为每个付费用户花多少钱？（含时间成本）—— atu 的红线
  是 ≤100 元，期望 <30 元
- **增长率红线**：每月最低希望涨多少？低于多少就不健康？

**worked example**：把"月活"作为北极星就完蛋——你会被诱导去做"涨粉但不
赚钱"的事。**北极星永远必须和钱直接挂钩**。

### 4. 可用资源

- 用户每天能给数字员工**多少分钟审稿时间**？（决定每天能发多少篇）
- 内容**预算**：是否能付小红书/知乎数据 API、能否付 LLM API、能否付
  设计/视频外包
- 用户**已经有的工具栈**：飞书 / 钉钉 / Notion / Cursor / Claude Code /
  生成式视觉工具
- 用户**已经在用**的 IM 和文档平台（决定 backend 选择）

### 5. 渠道与触点

- **信息源（入口）**：英文内容来源（Reddit / HN / Twitter / 外刊）、行业
  情报来源（竞品官网 / Product Hunt / 主流播客）
- **内容出口**：主线 / 侦察 / 储备的**三层渠道**模型（详见 Calibration）
- **触达管道**：用户主动进来的链路（站内 SEO / 平台搜索 / 社媒推荐 / 朋友
  推荐）

### 6. 红线与禁区

- 平台合规：AIGC 限流（小红书/知乎对纯 AI 内容降权）→ Charter 必须写
  "Agent 初稿必手改 ≥ 20%"
- 外链：小红书外链零容忍，知乎单篇外链≤1，公众号文末可放
- 营销密度：内容里直接提产品的比例上限（atu 设的是 1/5 = 20%）
- 矩阵号：单平台单账号还是多账号？多账号风险大（互相限流），第一阶段
  **坚决建议只 1 个号**
- 敏感话题：政治 / 医疗 / 金融 / 未成年——任何一项命中直接 reject

## Calibration 阶段：必须拍板的决议（按顺序）

### 决议 1：增长合伙人的 Structured Work 边界

> **重要**：这条只决定**结构化、循环性**的活——即哪些事进 Schedule、有
> Skill、产出落 Workbench。**用户和它 ad-hoc 对话（聊产品方向 / 增长策略 /
> 灵感讨论）不在这条边界里——那是默认全开的能力**（参见 SKILL.md 的
> "心智模型"第 3 层）。

**推荐 Structured work 范围**：**只做内容供应链**（原料筛选 → 半成品 →
成品 → 数据复盘）。**Structured 上不做**：客服回复、社群自动运营、付费
投放、PR、产品策略 PRD 撰写。

**为什么 structured 上做内容供应链**：
- 线性流水线 + 大量重复 + 单步可校验——最适合数字员工
- 客服/社群是"事件驱动 + 高情境"，AI 翻车成本高，做成 structured 任务
  会出事
- 付费投放/PR 是低频高决策，做成定时任务是错的频率
- **产品策略**虽然你随时可以 @ 它聊（ad-hoc），但**不应该有一个"每周一
  自动产出产品策略报告"的 Skill**——这种"定期开会式"产物会变成无效内容

**替代项**：如果用户**明确**已经有内容能力但缺数据分析，可以做
"数据分析合伙人"——但这是另一个 playbook，建议切换。

**反面校准**：用户经常会描述成"我希望它独当一面把增长负责起来，包括
跟我聊产品、聊灵感"——你不要把这些都翻译成 Skill。**结构化能干的活
就那一条线（内容供应链）；聊天能力是 free**。这条对齐没说清楚，
Configuration 会无意义地膨胀。

### 决议 1.5：Charter 上下文密度（role-specific 建议）

SKILL.md 通用决议要求拍板 Charter 密度档位。增长合伙人这个角色，**强烈
推荐 Standard 起步，并在以下字段写厚**：

| 字段 | 必写 | 为什么对 ad-hoc 对话很关键 |
|---|---|---|
| 产品一句话 | ✓ | 用户跟它聊产品方向时，它得知道你做的是什么 |
| 真实 ICP（含画像偏差） | ✓ | 聊"我们是不是该做 X 功能"时，它能用 ICP 校准 |
| 北极星 + CAC 红线 | ✓ | 聊任何增长想法时，它能用红线挡掉烧钱方案 |
| 主要竞品 + 差异化论据 | ✓ | 聊定位 / 价格 / 功能优先级时的参照 |
| 渠道分层（主线/侦察/储备/暂缓） | ✓ | 聊"要不要试 X 渠道"时直接对照 |
| 当前漏斗数据（注册→付费转化等） | 推荐 | 聊数据时它知道现状 |
| 产品路线图要点 | 视情况 | 聊产品讨论时减少重复同步 |

**反例**：Charter 只写了身份 + 北极星，用户随后跟它聊"我们是不是该砍掉
雅思学习场景"——它没有 ICP 描述，只能给空泛回答。这种情况下用户会觉得
"commando 也就那样"——其实是 Charter 写薄了。

**避免写进 Charter 的**：
- 短期工作日程（那是 Schedule 的事，不是 Charter）
- 复盘日志（那是 Episodic Memory 的事，由 Runtime 自动累积）
- 实时数据（那是 Connectors 拉的，不是 Charter 硬编码）

### 决议 2：内容公式（核心方法论）

**重要**：配比按用户当前 PMF 状态分三档。Discovery 阶段必须先确认 PMF 状态
再选配比，**不许直接照 PMF 后那套用在 pre-PMF 用户身上**。

#### 配比 A · Pre-PMF（< 10 付费用户）

| 公式 | 占比 | 角色 |
|---|---|---|
| **A. Bridge 公式** | 50% | 信源型，撑量+积累调性 |
| **B. 用户案例放大** | **30%** | 真实付费用户的故事/原话/反馈→内容→**让下一个同类自我识别** |
| **C. PLG 直接产品介绍** | 20% | 慎用，营销密度上限 |

**Pre-PMF 关键差异**：B 不是"爆款复刻"——而是**把每个付费用户的故事讲出来**。
这是 pre-PMF 阶段最高 ROI 的素材（用户少，但每个都珍贵）。

#### 配比 B · PMF 后早期（10-100 付费用户）

| 公式 | 占比 | 角色 |
|---|---|---|
| **A. Bridge 公式** | 50% | 仍是基本盘 |
| **B. 爆款复刻** | 30% | 此时可以开始追爆款概率 |
| **C. PLG 短内容** | 20% | 直接推产品 |

#### 配比 C · 规模化（100+ 付费用户）

**TBD**——atu 还没到这阶段，playbook 暂不给配比。
出现这种用户时 Onboarding 主动说"我没经验，咱们对照数据自己摸"，并把
你跑出来的经验沉淀回这份 playbook（PR 欢迎）。

#### 为什么必须按 PMF 状态分档

- Pre-PMF 阶段追爆款是错的——你没足够样本判断"爆"对你的 ICP 是不是真信号
- PMF 后阶段不放大用户故事是浪费——你已经验证过 ICP，故事就是放大器
- 不分档 → playbook 给出错的建议，Configuration v0.1 写错

**真实测试印证**（2026-06-11 commando onboarding dry-run）：测试案例处
pre-PMF（个位数 paid），Onboarding 给出配比 A，把"爆款复刻 30%"换成
"用户案例放大 30%"。如果按旧版统一 50/30/20 写 Configuration，**用户 A
那种关键素材将不会被系统性使用**。

**替代项**：如果用户产品**视觉极强**（设计工具 / 创作工具），任一档位的
A 可以扩到 60-70%，A 改成"视觉演示型 Bridge"——但仍然必须是 3 公式而非
单一。

### 决议 3：渠道分层

**推荐三层模型**：

| 层级 | 角色 | 选什么 |
|---|---|---|
| **主线** | 增长主战场，频率高 | **只选 1 个**，一般是小红书或知乎 |
| **侦察** | 复用主线内容做小成本测试 | TikTok / X / 公众号转载 |
| **储备-信息源** | 不发布，只采集原料 | Reddit / HN / 外刊订阅 |
| **储备-SEO** | 月频长尾 | 知乎/独立站博客 |
| **暂缓** | 明确**不做**的列表 | LinkedIn / YouTube Shorts / 付费投放 |

**关键**：必须明确"暂缓"列表——否则用户会贪心铺渠道，结果每个都做不深。

**升级规则**：侦察→主线的条件**写进 Charter**——atu 用的是"连续 3 周
单篇互动 > 历史均值 5 倍 OR 单篇带来注册 > 20"。

### 🚨 决议 4 之前的强制前置检查：归因可观测性

**在确定 Schedule 之前，必须问用户**：你能否回答"上个月每个新增付费
用户来自哪个渠道 / 哪条具体内容"？

如果用户答**不能** → **归因黑箱**——这是 Configuration **之外**的前置阻塞。
所有"复盘 / 数据回收 / ROI 分析"类的 task 在归因装好之前**都会跑空**。

必须把以下作为 **Configuration 之外的 prerequisite todo** 写进最终的
`my-agent/README.md`：

- 部署 [Plausible](https://plausible.io/) / [Umami](https://umami.is/) /
  [简悦](https://www.matomo.org/) 等轻量分析工具
- 注册流程加 "你从哪里知道我们的" 选项
- 所有发布出去的链接（小红书简介 / 知乎文末 / 邮件签名）打 UTM

**为什么这是 must-have 而非建议**：

- 归因黑箱 → 不知道哪条内容/渠道有效 → 增长合伙人的复盘任务返回的全是
  "数据不可知"
- 这件事数字员工**没法替用户做**（涉及产品代码改动 + 第三方服务接入）—— 
  Onboarding 必须明确告诉用户"这是你自己的活，不是我能代劳的"
- 漏掉这条 → Configuration 表面跑通，但 weekly_data_review 永远报空

**真实测试印证**（2026-06-11）：测试案例出现"用户 B 凭空在支付后台出现"
——一个 paid user 来源完全未知。Onboarding 主动标为前置阻塞，写进
my-agent README 的最高优先级 todo。

### 决议 4：任务节奏（Schedule 雏形）

**推荐日常班次**（atu v0.5 跑通的）：

| 时间 | 任务 | 输出 |
|---|---|---|
| 每日 08:22 | 信息源扫描 + LLM 打分 | 选题池入库 |
| 每日 11:00 | 主公式（A）笔记初稿 | Inbox → 待审稿 |
| 每日 11:30 | 自检（品牌调性 + 合规） | 自检报告 |
| 每日 14:00（周二/四） | 公式 B 视频脚本 | Inbox → 待审稿 |
| 每日 16:00 | 效果分析 | ROI 复盘 |
| 每周一 09:30 | 周度热点深扫 | 选题机会矩阵 |
| 每月初 | SEO 长文 | 知乎/博客草稿 |
| 每周日 19:00 | 周报 + 更新 Semantic Memory | 反思文档 |

**Calibration 时该问的**：用户每天的**审稿时段**是什么？把数字员工的产出
节点对齐到用户能审稿的时段——atu 是上午 11:00 出稿 + 12:00 午饭时审稿，
因为 Qihang 中午有时间。

### 决议 5：Structured Skill 启用清单

> 再次强调：这里列的是**结构化任务**用的 Skill（有 trigger、有产出、入
> Workbench）。用户和它聊产品/灵感这种 ad-hoc 用途**不需要在这里列**——
> 那部分由 Charter 密度（决议 1.5）支撑。

**通用 Skill 推荐（来自 atu 的 8 个 marketing kit）**：

- ✅ 营销文案（写主公式内容）
- ✅ 营销效果分析（每日 ROI 复盘）
- ✅ 竞品追踪
- ✅ 社媒热点追踪
- ✅ 广告合规检查
- ✅ 品牌调性审查
- ✅ SEO 内容优化
- 储备：活动策划

**自定义 Skill（按优先级）**：

- 🔧 `<platform>-<formula>-bridge`（如 `xhs-bilingual-bridge`）：把信息源
  原料加工成主平台的成品格式。**这是 commando 用户最值得自己写的 Skill**。
- 🔧 `<source>-source-mining`：每日扫指定信息源 + LLM 4 维打分（ICP / 钩子 /
  红线 / 多样性）。
- 🔧 `<platform>-video-script-cloner`：从爆款视频提结构 + 生成自家版本。

**Skill Registry 查询时机**：
- 自定义 Skill 的每一项都**先**问 Registry：是否有 `@<author>/<similar>`
  可以 import？
- 在 atu 沉淀阶段（即未来），上述 3 个自定义 Skill 应该作为 `@frank-pu/*`
  在 Registry 出现，用户可以 fork 改。

### 决议 6：Workbench backend

**推荐**：如果用户**已经在用飞书**，强烈推荐 Feishu backend。原因：
- bitable 原生支持看板/日历/甘特/表格多视图，无需自建 UI
- Wiki 适合做章程/复盘报告的展示层
- IM 卡片做"任务状态推送"原生体验好，移动端可用
- lark-cli 已开放绝大部分 OpenAPI，agent 友好

**替代项**：
- 用户用 Notion → Notion backend（视图能力略弱但够用）
- 用户什么都不用 → Local backend + `commando dashboard`（仅推荐过渡期）
- 用户用钉钉/企微 → 等社区 backend driver，或让 agent 当场生成

### 决议 7：红线明文化（必须落进 Charter）

最常被忽略、出问题最大的几条：

- AIGC 限流应对：**Agent 初稿必手改 ≥ 20%**
- 外链：明文写各平台的外链规则
- 营销密度上限：明文写百分比
- 矩阵号：明文写"第一阶段坚决只 N 个号"
- 复刻视频边界：**保留创意结构，不引用原文字/标题/截图**
- 工作流硬 Gate：**所有发布必须用户点确认**——这条不许折中

## 常见反模式（用户会问，你要果断拒绝）

| 用户会说 | 你要回答 |
|---|---|
| "能不能让 agent 自动发布，不用我审？" | 不能。发布 Gate 是 commando 的红线设计，不为任何用户绕过。Charter 红线之一。 |
| "我能不能开 10 个矩阵号让 agent 同时运营？" | 不行，平台矩阵号检测越来越严。第一阶段单号深耕，单号验证 3-6 个月后再说。 |
| "agent 写得不够'惊艳'，能不能换更强的模型？" | 不是模型问题，是 Configuration 问题——重新校准 Charter / Skill 描述 / few-shot 标杆。 |
| "我可以不要 Charter，直接让 agent 看着办吗？" | 不可以。Charter 是 commando 的宪法层，没有 Charter 等于让 agent 每天醒来失忆；ad-hoc 对话也会失去接地材料。 |
| "AIGC 那 20% 手改太麻烦了能不能跳过？" | 不能跳过——这条直接对应平台流量。要么手改 20%，要么接受流量被压一半。 |
| "我想让它每周自动写一份'产品策略报告'交给我" | 大概率是错的需求。产品策略是低频高情境决策，做成定时产物只会变成你不读的废件。**正确做法**：你想聊时随时 @ 它（ad-hoc），它用 Charter 上下文跟你聊。 |

## v0.1 应该长什么样（给 Onboarding 一个交付物的具体标杆）

最后产出的 Configuration 目录，**charter.md 至少包含**：

- 角色身份（名字 + 画像）
- 服务产品（一句话）+ 真实 ICP 描述
- 北极星 + 次级指标 + CAC 红线 + 90 天目标
- 3 个内容公式 + 比例
- 渠道分层（主线 / 侦察 / 储备 / 暂缓）
- 红线 7 条

**schedule.yaml 至少包含**：上面"决议 4"那张表的 7-8 个任务声明。

**skills/ 至少包含**：8 个通用 marketing skill（可全部 from Registry import）+
1 个自定义 `<platform>-<formula>-bridge` Skill 占位（status: draft）。

**connectors.yaml 至少包含**：选定的 backend + 信息源（Reddit / 外刊 RSS 等）
+ AI 模型路由（默认 Claude，备用 OpenRouter 兼容）。

如果产出的 Configuration 缺任何一项，Onboarding skill 必须主动补问。

## Re-onboarding 时这份 playbook 怎么用

- 用户跑 2 周后回来，先看他的 KPI 数据 + Episodic Memory
- 如果出现这些信号，**主动建议改 Charter**：
  - 真实付费用户画像 ≠ 当初 ICP → 重写 ICP 段
  - 某公式连续 5 篇低于均值 → 砍掉或重新校准
  - CAC 突破红线 → 暂停高成本任务，回到信息源筛选标准
  - 主线渠道严重限流 → 升级一个侦察渠道顶上
- **不主动改的事**：用户没数据支持的"我感觉不太对"——记录到
  `.onboarding/issues.md`，留待下次。
